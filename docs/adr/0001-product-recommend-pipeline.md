# ADR-0001：产品推荐采用检索管线而非 LLM 生成

**日期**：2026-07-19  
**状态**：已接受

## 背景

母婴垂类 Agent 需要为企业员工推荐本企业产品。产品推荐的核心业务约束是"严禁幻觉"——推荐的产品必须 100% 来自企业产品库。传统 LLM 直接生成产品推荐存在编造不存在的产品名、品类名的幻觉风险，prompt 约束无法根本杜绝此问题。

## 决策

产品推荐走 **检索 → 排序 → 输出** 硬管线：

1. **意图分类**：LLM structured output，枚举约束
2. **宝宝需求提取**：LLM structured output，固定 schema
3. **产品检索**：纯向量检索 + 标签过滤，从企业产品库中检索候选产品，排序取 Top-N
4. **回复组装**：产品名从数据库取值拼入回复模板，LLM 仅负责生成推荐理由的解释性文字

品类名映射：LLM 提取的品类名先做向量匹配映射到企业产品库中的实际品类，防止品类名幻觉。

## 备选方案

**方案 B**：全链路 LLM 生成推荐，用 prompt + RAG 约束。  
风险：LLM 本质是概率生成，无法 100% 保证产品名不编造。

**方案 C**：全链路硬管线，包括意图识别也用规则+向量匹配。  
缺陷：自然语言理解能力弱，复杂表述（如"宝宝老是拉，屁股都红了，之前喝牛奶也过敏"）难精确解析。

## 后果

- **优点**：产品名零幻觉，业务约束绝对满足；LLM 被限制在"理解 + 解释"角色，不参与产品选择
- **代价**：架构复杂度增加——需要维护三条并行管线（产品推荐 / 膳食健康咨询 / 拒答），意图路由故障会影响下游正确性
- **缓解**：意图分类用 LLM structured output + 枚举约束，降低误判概率
