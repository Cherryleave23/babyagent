# 母婴垂类 B2B 智能 Agent 规范

面向母婴企业门店员工的 AI 助手，充当"智能儿科医生 + 儿童营养师"角色，通过微信交互，辅助为 0-6 岁宝宝量身定制膳食方案、推荐本企业产品、提供儿科健康咨询。

## 需求（Requirements）

### 意图识别与路由
- R1（无处不在型）：系统应当对每条员工消息执行意图分类，输出为 `product_recommend | diet_advice | health_consult | out_of_scope` 之一。

### 宝宝需求提取
- R2（状态驱动型）：当意图为 `product_recommend`，系统应当从员工消息和当前宝宝档案中提取结构化需求（月龄、症状/关注点、饮食禁忌、所需产品品类），输出为固定 schema。

### 产品推荐防幻觉
- R3（无处不在型）：产品推荐流程中，最终输出的产品名称必须直接取自企业产品库，不得经 LLM 生成。
- R4（无处不在型）：当 LLM 提取的品类名无法匹配到企业产品库中任何品类时，系统应当做向量匹配映射到最近库内品类，而非使用 LLM 原始输出。

### 宝宝档案管理
- R7（无处不在型）：宝宝档案核心字段（姓名、性别、出生日期、过敏史、饮食禁忌）的创建与修改必须通过指令解析（规则驱动），不得经 LLM 自由生成写入。
- R8（事件驱动型）：当员工发送建档/更新档案指令（如 `@Agent 建档 小明 男 2025-03-15`、`@Agent 小明 过敏史添加 花生`），系统应当解析指令并写入对应字段。
- R9（状态驱动型）：LLM 在需求提取和咨询回答时，可以读取当前宝宝档案作为上下文，但不得直接修改档案核心字段。
- R10（无处不在型）：系统应当维护每个宝宝档案的生长记录，字段为 {日期, 身高, 体重} 的追加列表，通过指令驱动更新。

### 备注机制
- R11（事件驱动型）：每轮对话结束时，LLM 应当判断对话中是否有值得记录的宝宝特征信息（如新发现的过敏反应、饮食偏好、特殊情况），如有则自动生成一条备注候选内容。
- R12（无处不在型）：备注写入后系统应当向员工推送简短通知（"💾 已为{宝宝名}记录：{备注内容}"），无需员工回复即默认生效，员工可通过回复"撤销"或手动编辑来回滚。备注记录须保留来源对话 ID 供追溯。

### 问题域边界
- R5（无处不在型）：系统应当对意图为 `out_of_scope` 的消息直接返回拒答模板，不调用 LLM 生成回复。
- R6（事件驱动型）：当识别到急重症症状描述（如高热惊厥、呼吸困难、意识模糊等），系统应当拒绝提供建议并引导立即就医。

### 基座模型
- R13（无处不在型）：系统应当支持 DeepSeek、GLM（z.ai）、ChatGPT（OpenAI）三种基座模型，通过配置文件切换，运行时不重启。
- R14（无处不在型）：系统应当基于 Hermes 的 `ProviderProfile` 抽象层，保留全部 33 个 model-provider 插件，不做精简。

### Web 管理控制台
- R15（无处不在型）：系统应当提供一个 Web 管理控制台，供门店 IT 管理员进行初始配置与运维管理。
- R16（无处不在型）：Web 控制台应当支持以下功能：绑定/解绑微信 Clawbot（扫码）、配置基座模型 API Key、管理已授权的员工微信列表、查看宝宝档案列表。
- R17（无处不在型）：Web 控制台的聊天功能仅为可选——员工交互的主通道是微信，Web 聊天仅作运维调试用途。

### 宝宝快速切换
- R20（事件驱动型）：当员工消息中可识别出目标宝宝名（隐式切换），系统应当自动将当前活跃会话的宝宝指针切换到该宝宝，并在回复中附简短确认提示（"🔄 已切换到{宝宝名}的档案"）附带档案摘要。
- R21（事件驱动型）：当员工发送显式切换指令（如 `@切换宝宝 小明`、`@宝宝 小红`），系统应当强制切换活跃宝宝指针。
- R22（无处不在型）：当消息中未包含可识别宝宝名时，系统应当沿用上一轮对话的活跃宝宝，不做切换。
- R23（不期望型）：如果员工消息中的宝宝名在同一门店下存在同名冲突，系统应当追问区分信息（出生日期、性别），确认后再切换。

### 技能自进化（保留 Hermes Skills）
- R18（无处不在型）：系统应当保留 Hermes 的 Skills 机制（bundled/managed/workspace），支持技能的自创建、自改进、跨会话复用。
- R19（无处不在型）：系统应当保留 Hermes 的学习回路（learning graph + curator），使 Agent 能从母婴咨询实践中积累专业经验。

### 端侧统一数据库（三层合并模型）
- R24（无处不在型）：端侧数据库应当合并三层数据为统一检索空间——层A（总部种子库，只读）、层B（企业拓展库，企业自录入）、层C（运行时数据，宝宝档案/对话/技能记忆）。
- R25（无处不在型）：总部种子库对端侧只读，不得由端侧修改。
- R26（无处不在型）：系统应当定时（默认每日）从云端拉取总部种子库版本号，有更新时拉取增量 delta，仅重建变动条目的向量；当 delta 超过 30% 条目或 schema 变更时触发全量重建。
- R27（无处不在型）：总部种子库同步失败不得影响服务可用性——系统应当继续使用本地已有缓存副本，并在 Web 控制台告警。
- R28（无处不在型）：初始部署包应当包含预构建向量索引的种子知识库，使首次部署启动时间不超过数分钟。
- R29（无处不在型）：向量检索应当在三层数据合并空间中执行，但检索结果不得向员工暴露数据来源层标识。

### 上下文压缩与跨 Session 宝宝档案同步
背景：多员工 session 严格隔离，摘要不得跨 session 传播（防止信息噪声与责任混淆）。但多个员工共同服务于同一个宝宝，宝宝状态的增量变化（如新发现的过敏信息）必须跨 session 共享。

- R30（无处不在型）：上下文压缩应当输出两部分结构化内容——（a）session 私有摘要（仅本 session 可用），（b）宝宝档案变更建议（跨 session 共享）。
- R31（无处不在型）：压缩 LLM 的输入应当包含当前 session 对话历史 + 当前宝宝档案快照，使其能够判断哪些是新信息、哪些是重复信息。
- R32（无处不在型）：宝宝档案变更操作必须为结构化指令（append / replace / delete），由系统执行，LLM 不得直接覆写整个档案。
- R33（无处不在型）：仅 `confidence: high` 的变更自动生效；低置信度变更写入待确认队列，供后续员工人工核实。
- R34（无处不在型）：关键字段（过敏信息）的变更操作必须使用 `append`（只增不删），删除操作只能由人工标记。
- R35（无处不在型）：每次宝宝档案写入操作必须记录审计日志（来源 session_id、时间、变更内容），审计日志与备注溯源合并为统一变更历史表。
- R36（无处不在型）：宝宝档案更新接口应当使用乐观锁或时间戳 Last-Write-Wins 策略处理并发写入，冲突时记录冲突日志。
- R37（事件驱动型）：当员工 B 进入已存在宝宝的会话且当前无活跃宝宝指针时，系统应当从最新宝宝档案中生成简洁的"宝宝当前状态"摘要（结构化档案内容，非其他员工对话摘要），注入到会话首条系统消息中。
- R38（无处不在型）：宝宝指针在对话中途被切换回该宝宝时，状态注入发生在切回后第一次 LLM 调用前，不得在每轮对话中重复注入。
- R39（无处不在型）：压缩执行顺序应为（1）先执行宝宝状态提取+更新，（2）再基于更新后的档案生成 session 私有摘要注入当前 prompt。步骤（1）可异步但若当前 session 需立即使用新状态则同步执行。

### 产品推荐空结果降级
- R40（不期望型）：如果产品检索管线返回空结果，系统应当明确告知员工"未找到匹配产品"，同时仍可提供与该需求相关的健康/膳食建议（来自知识库 RAG），不得降级为 LLM 自由生成产品推荐。

### 部署模型
- R41（无处不在型）：系统应当采用一户一实例的部署模型——每个企业独立部署一个 Agent 实例，包含该企业专属的产品库、员工账号、宝宝档案。不同企业的实例之间物理隔离，产品库绝对不交叉。
- R42（无处不在型）：一个实例可服务该企业下的多个门店，通过绑定多个微信 Clawbot 账号区分不同门店。
- R43（无处不在型）：同一实例内的多门店数据共享策略应当由企业 IT 管理员在 Web 控制台中可调配——可选项包括：（a）全部共享（产品库+员工+宝宝档案全企业通），（b）仅共享产品库（员工和宝宝档案按门店隔离），（c）完全隔离（每个门店独立数据空间，仅产品库共享）。默认选项为（a）。

## 边界（Boundaries）

### 范围内
- 0-6 岁宝宝的儿科健康问题咨询（常见症状评估、生长发育建议、疫苗接种指导，仅限身体层面）
- 0-6 岁婴幼儿营养与膳食方案（辅食添加、食谱定制、过敏规避、营养素评估）
- 基于企业产品库的产品推荐（必须在库内，严禁幻觉）
- 微信 Clawbot（iLink 协议）作为唯一的员工交互通道
- Web 管理控制台（运维配置：微信绑定、API Key、员工管理、宝宝档案）
- 基于 Hermes 内核魔改，保留 Gateway + Session + Model Provider + Skills + Learning Loop
- Hermes 全部 33 个 model-provider 插件

### 范围外（不做）
- 成人疾病 / 妇产科相关问题
- 急重症诊断（法律风险，须引导就医）
- 0-6 岁宝宝的心理/行为问题（睡眠训练、早教、行为引导等育儿护理）
- 非婴幼儿食品推荐
- 超出母婴垂类范围的一切问题（直接拒答）
- Hermes 的 Desktop App（Electron）、TUI、Voice Wake、Talk Mode、Browser Control、Canvas/A2UI
- 微信以外的消息通道（WhatsApp、Telegram、Slack、Discord 等全部砍掉）
- 图片/视频生成、语音转录/TTS、Spotify、Google Meet、Kanban 等无关插件

## 不变式（Invariants）

- I1：产品推荐管线中，最终回复中的产品名称集合必须是企业产品库产品名称集合的子集。  
  > 自检：`recommended_products ⊆ db_products; return recommended_products.all? { |p| db_product_names.include?(p) }`
- I2：`out_of_scope` 意图的消息不得调用基座 LLM 生成回复内容。  
  > 自检：`return !llm_called_for(intent: :out_of_scope)`
- I3：备注内容不得参与产品推荐决策——产品检索管线的输入不得包含备注字段。  
  > 自检：`return product_search_inputs.exclude?(:notes)`
- I4：每个会话在任何时刻最多有一个活跃宝宝指针，员工回复的目标宝宝必须可追溯到已建档宝宝列表。  
  > 自检：`return session.active_baby.nil? || store_babies.exists?(session.active_baby)`
- I5：总部种子库（层A）对端侧只读——任何端侧操作不得修改层A数据。  
  > 自检：`return seed_db.write_attempts.empty?`
- I6：总部种子库同步进行中或同步失败时，端侧服务不得中断——始终使用本地已有缓存应答。  
  > 自检：`return available_during_sync() == true`
- I7：session 私有摘要不得跨 session 传播——任何供 LLM 使用的上下文不得包含其他 session 的摘要内容。  
  > 自检：`return cross_session_context.content_includes?(other_session_summaries) == false`
- I8：宝宝档案的过敏信息字段必须使用 append-only 策略——LLM 发起的删除操作不被执行，仅人工标记可删除。  
  > 自检：`return baby.allergy_changes.all? { |c| c.operation != :delete || c.source == :human }`
- I9：产品检索管线返回空结果时，不得降级为 LLM 生成产品推荐——空结果就是空结果。  
  > 自检：`return !(product_search.empty? && reply.contains_product_recommendation?)`

## 验收标准（Acceptance）

### 意图分类与拒答
- **AC1**：给定一条超出母婴范围的消息（如"推荐一只股票"），当系统执行意图分类，那么输出为 `out_of_scope`，且最终回复为预设拒答模板，未调用 LLM 生成回复内容。
- **AC2**：给定一条含急重症描述的消息（如"宝宝高热惊厥、意识模糊"），当系统识别意图，那么回复引导立即就医，不提供任何医疗建议。

### 产品推荐防幻觉
- **AC3**：给定宝宝档案（8个月、牛奶蛋白过敏）和员工消息"该用什么产品"，当系统走完产品推荐管线，那么最终回复中所有产品名均存在于企业产品库中。
- **AC4**：给定 LLM 提取的品类名为库内不存在的"湿疹特效霜"，当系统执行品类映射，那么品类被映射到库内最近品类"婴儿护肤霜"，不使用 LLM 原始品类名。
- **AC5**：给定产品检索返回空结果，当系统生成回复，那么回复告知"未找到匹配产品"并提供健康建议，不包含 LLM 编造的产品名。

### 宝宝档案
- **AC6**：给定员工发送 `@Agent 建档 小明 男 2025-03-15`，当系统解析指令，那么创建档案包含姓名=小明、性别=男、出生日期=2025-03-15，月龄自动计算。
- **AC7**：给定员工发送 `@Agent 小明 过敏史添加 花生`，当系统解析指令，那么小明的过敏史列表追加"花生"，不覆盖已有条目。
- **AC8**：给定对话中员工提到"小明最近吃鸡蛋好像也有点反应"，当本轮对话结束，那么系统自动生成备注候选并推送通知，备注不参与后续产品推荐检索。

### 宝宝快速切换
- **AC9**：给定会话当前活跃宝宝为小明，当员工发送"小红湿疹好点了吗"，那么系统自动切换到小红并回复确认提示（含档案摘要）。
- **AC10**：给定门店下存在两个同名"小明"（出生日期不同），当员工发送"小明的过敏情况"，那么系统追问出生日期或性别以区分。
- **AC11**：给定员工连续发送"宝宝该吃什么"未提及宝宝名，那么系统沿用上一轮的活跃宝宝，不做切换。

### 跨 Session 宝宝档案同步
- **AC12**：给定员工 A 的会话中 LLM 从对话提取了宝宝过敏新信息（confidence: high），当压缩执行完成，那么该信息作为宝宝档案变更已生效，员工 B 在下次访问该宝宝时能在状态注入中看到更新。
- **AC13**：给定员工 A 的 session 私有摘要包含主观判断，当员工 B 进入同一宝宝的会话，那么 B 的上下文中不包含 A 的私有摘要内容。
- **AC14**：给定两个员工同时写入同一宝宝档案的不同字段，当并发发生，那么系统按时间戳 Last-Write-Wins 处理，冲突写入审计日志。

### 员工隔离
- **AC15**：给定同一门店的两个员工分别与 Agent 对话，当员工 A 发送消息后员工 B 发送另一消息，那么两次对话的上下文完全独立，互不可见。
- **AC16**：给定员工 A 正在对话，当员工 B 首次进入已存在宝宝的会话且无活跃宝宝指针，那么系统注入"宝宝当前状态"摘要到 B 的首条系统消息。

### 多企业部署
- **AC17**：给定企业 X 的实例，当员工查询产品，那么返回的产品均来自企业 X 的产品库，不包含任何其他企业产品。
- **AC18**：给定管理员在 Web 控制台将数据共享策略从"全部共享"改为"仅共享产品库"，当员工 A（门店 1）尝试搜索宝宝，那么只能搜索到门店 1 的宝宝档案。
